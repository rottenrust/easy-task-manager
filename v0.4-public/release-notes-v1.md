# Release Notes v1

## TM-001 — Спроектировать новую модель задачи

Реализована новая структура задачи с полями:
- `id`
- `text`
- `status`
- `created_at`
- `completed_at`

### Что изменено

- Обновлена нормализация задач в `task_manager/storage.py`:
  - Поддерживается чтение новой модели.
  - Добавлена обратная совместимость со старым полем `done` при чтении существующих данных.
  - Добавлены статусные константы `STATUS_ACTIVE` и `STATUS_COMPLETED`.
- Обновлена CLI-логика в `task_manager/cli.py`:
  - `add` теперь создает задачи с новой моделью и `created_at` в UTC ISO-формате.
  - `done` переводит задачу в статус завершенной и заполняет `completed_at`.
  - `list` отображает состояние на основе поля `status`.

### Примечание

В рамках TM-001 структура данных введена и используется на уровне CLI/хранилища. Отдельная бизнес-логика архивирования и финализация статусной модели покрываются следующими задачами бэклога (TM-002, TM-003 и далее).

### Review follow-up

- Усилена валидация `status` при загрузке: принимаются только валидные значения (`active`, `completed`).
- Улучшена нормализация `completed_at` для legacy-записей: для завершенных задач без корректного `completed_at` используется `created_at` (если есть), иначе пустая строка как fallback.

### QA follow-up

- Выполнен функциональный smoke новой модели:
  - `list` на пустом наборе.
  - `add` двух задач.
  - `done` одной задачи.
  - `delete` второй задачи.
  - Проверка структуры `tasks.json` после операций.
- Выполнен регрессионный тест совместимости со старым форматом:
  - Загрузка задач с полем `done`.
  - Корректное отображение в `list`.
  - Успешный переход legacy-задачи в новый формат после `done`.
- Блокирующих багов не выявлено.

## TM-002 — Добавить статусы задач: активная и архивная

Статусная модель обновлена до двух бизнес-статусов:
- `active`
- `archived`

### Что изменено

- Обновлено хранилище в `task_manager/storage.py`:
  - Заменен внутренний статус `completed` на `archived`.
  - Добавлена обратная совместимость при чтении: legacy-статус `completed` автоматически нормализуется в `archived`.
  - Сохранена совместимость со старым полем `done` (`true` -> `archived`, `false` -> `active`).
- Обновлена CLI-логика в `task_manager/cli.py`:
  - Команда `done` теперь переводит задачу в статус `archived`.
  - Обновлены тексты CLI-сообщений под терминологию архивирования.
  - Отображение `list` синхронизировано с новым статусом `archived`.

### Примечание

В рамках TM-002 выполнена стабилизация статусной модели на уровне CLI/хранилища. Поведение отдельного представления активного списка и архива по разным командам остается задачей TM-003 и следующих задач сервисного слоя.

### Review follow-up

- Убрана "магическая строка" legacy-статуса в `task_manager/storage.py`: добавлена константа `LEGACY_STATUS_COMPLETED` для явной и безопасной нормализации старых данных.
- Синхронизирован CLI-help с фактическим поведением TM-002: `done` теперь описан как архивирование задачи (`Archive a task`), что снижает терминологическую путаницу.

### QA follow-up

- Выполнен функциональный smoke TM-002 через CLI:
  - `add` новой активной задачи.
  - `list` с проверкой отображения активной задачи как `[ ]`.
  - `done` с переводом задачи в архив (`archived`).
  - Повторный `done` с проверкой корректного сообщения `already archived`.
  - `delete` тестовой задачи и повторная проверка `list`.
- Выполнен регрессионный тест совместимости форматов `tasks.json`:
  - `status: completed` корректно нормализуется в `archived`.
  - legacy-поле `done: true/false` корректно нормализуется в `archived/active`.
  - После `done` для legacy-активной задачи статус и `completed_at` корректно обновляются и сохраняются в новой модели.
- Блокирующих багов не выявлено.

## TM-003 — Реализовать архивирование выполненной задачи вместо отображения ее в активном списке

Цель TM-003: отделить представление активных задач от архива на уровне CLI.

### Что изменено

- Выполнена developer-декомпозиция перед реализацией:
  - Подтвержден scope: команда `list` показывает только активные задачи.
  - Зафиксировано, что отдельный просмотр архива (команда/флаг) не входит в TM-003.
  - Определены проверки: функциональный smoke для нового поведения + регрессия `done` и `delete`.
- Обновлена CLI-логика в `task_manager/cli.py`:
  - `list` теперь фильтрует задачи по статусу `active`.
  - Архивные задачи (`archived`) больше не выводятся в активном списке.
  - При отсутствии активных задач сохраняется текущее сообщение `No tasks yet.`
- Обновлена документация в `README.md`:
  - Описание команды `list` уточнено до "показать список активных задач".

### Review follow-up

- Проверена согласованность со статусной моделью TM-002:
  - `done` продолжает корректно переводить задачу в `archived` и выставлять `completed_at`.
  - Новая фильтрация `list` не затрагивает запись/хранение архивных задач, а меняет только отображение активного набора.
- Оценены риски регрессии:
  - Риск потери данных отсутствует, так как формат хранения не менялся.
  - Изменение локализовано в read-only сценарии отображения списка.

### QA follow-up

- Выполнен функциональный smoke CLI:
  - `list` на пустом наборе -> `No tasks yet.`
  - `add` двух задач -> обе видны в `list`.
  - `done` одной задачи -> в `list` остается только активная.
  - `done` второй задачи -> `list` возвращает `No tasks yet.`
- Выполнен регрессионный smoke:
  - Повторный `done` для архивной задачи возвращает `already archived`.
  - `delete` удаляет задачу по ID независимо от статуса.
  - `tasks.json` сохраняет архивные задачи в `status=archived`, несмотря на отсутствие в `list`.
- Блокирующих багов не выявлено.

## TM-005 — Обновить JSON-хранилище под новую модель задачи

Цель TM-005: закрепить хранение задач в `tasks.json` в каноничном формате новой модели (`id`, `text`, `status`, `created_at`, `completed_at`) независимо от входного формата данных.

### Что изменено

- Выполнена developer-декомпозиция перед реализацией:
  - Проверен текущий scope: сервис/CLI уже используют новую модель, но запись в файл происходила без повторной нормализации.
  - Определен риск: при прямом вызове `save_tasks()` в файл могли попасть неканоничные структуры.
  - Выбрано минимальное решение: единая функция нормализации задачи и обязательная нормализация перед сохранением.
- Обновлено хранилище в `task_manager/storage.py`:
  - Добавлена `_normalize_task(item)` для единой валидации и нормализации записи задачи.
  - `load_tasks()` переведен на переиспользование `_normalize_task()` вместо дублирования логики.
  - `save_tasks()` теперь всегда нормализует входной список и сохраняет только валидные задачи в новом формате.

### Review follow-up

- Проверена консистентность read/write-логики:
  - Одинаковые правила нормализации применяются и при чтении, и при записи.
  - Для активных задач принудительно сохраняется `completed_at: null`; для архивных корректно вычисляется fallback из `created_at`.
- Оценена безопасность изменений:
  - Изменения локализованы в `storage.py` и не меняют контракт CLI.
  - Влияние на обратную совместимость сохранено: legacy-поля (`done`, `status=completed`) продолжают корректно преобразовываться.

### QA follow-up

  - Через `save_tasks()` записан смешанный набор (legacy `done`, валидная новая запись, невалидная запись).
  - Проверено, что в `tasks.json` сохранены только валидные задачи в каноничном формате новой модели.
- Выполнен регрессионный smoke CLI:
  - `add` -> `list` -> `done` -> `list` -> `delete` -> `list`.
  - Подтверждено, что пользовательское поведение команд не изменилось.

## TM-006 — Добавить миграцию старого формата `tasks.json` к новой модели

Цель TM-006: выполнять не только чтение legacy-формата, но и его автоматическую фиксацию в каноничном формате новой модели.

### Что изменено

- Выполнена developer-декомпозиция перед реализацией:
  - Проверен текущий status quo: `load_tasks()` нормализует legacy-записи в памяти, но не закрепляет миграцию в `tasks.json`.
  - Определен риск: один и тот же legacy-набор продолжает мигрироваться при каждом чтении, а файл остается устаревшим.
  - Выбран минимальный подход: при обнаружении любых legacy/неканоничных записей выполнять одноразовую автозапись уже нормализованного списка.
- Обновлено хранилище в `task_manager/storage.py`:
  - В `load_tasks()` добавлен флаг `migration_needed`.
  - Миграция помечается, если хотя бы одна запись изменилась после `_normalize_task()` или если встречены невалидные записи, отброшенные при нормализации.
  - При `migration_needed=True` вызывается `save_tasks(normalized)`, что фиксирует `tasks.json` в новой модели.
- Обновлен `README.md`:
  - Добавлено описание автоматической миграции legacy-формата (`done` / `status=completed`) при чтении.

### Review follow-up

- Проведено senior-level ревью внесенных изменений:
  - Подтверждена идемпотентность миграции: после первой автозаписи повторные чтения не вызывают лишних перезаписей, так как формат уже каноничный.
  - Проверена локальность изменений: модифицирован только read-path хранилища; контракты сервисного слоя и CLI не изменены.
  - Оценен риск рекурсии: `load_tasks()` вызывает `save_tasks()`, но `save_tasks()` не вызывает `load_tasks()`, цикл отсутствует.

### QA follow-up

- Выполнен целевой тест миграции TM-006:
  - В `tasks.json` записан legacy-набор (`done`, `status=completed`) и невалидная запись.
  - Выполнен `python3 -m task_manager.cli list`, что триггерит `load_tasks()`.
  - Проверено, что файл автоматически переписан в новую модель (`status=active|archived`, `completed_at` согласован, невалидная запись удалена).
- Выполнен регрессионный smoke CLI после миграции:
  - `add` новой задачи.
  - `done` добавленной задачи.
  - `delete` добавленной задачи.
  - `list` отрабатывает штатно и показывает только активные задачи.
- После QA исходный `tasks.json` восстановлен из локального бэкапа.
- Блокирующих багов не выявлено.
