# Public Release Workflow (`v0.4-public`)

Этот документ описывает ручной процесс публикации деперсонализированной версии проекта в публичную ветку.

## Что публикуется

- Публикуется только папка `v0.4-public`.
- Runtime-файлы в составе `v0.4-public` обязательны и хранятся как образцы:
  - `v0.4-public/task_manager.log`
  - `v0.4-public/tasks.json`
  - `v0.4-public/tasks.json.bak`

Важно: содержимое этих файлов должно быть только учебным (sample data only).

## Обязательная санация перед пушем

Проверить, что в `v0.4-public`:
- нет токенов, секретов и приватных ключей;
- нет локальных путей (`/Users/...`, `C:\\Users\\...`);
- нет реальных user id и журналов использования.

Команды проверки:

```bash
cd /path/to/project
rg -n --hidden --glob '!.git/*' -e '/Users/' -e 'C:\\Users\\' -e '[0-9]{8,12}:[A-Za-z0-9_-]{30,}' -e 'sk-[A-Za-z0-9]+' v0.4-public
rg -n --hidden --glob '!.git/*' -e '123456789' -e '987654321' v0.4-public
python3 -m unittest discover -s tests -p 'test_*.py' -q
```

## Git-стратегия (same repo + public branch + rewritten history)

Ниже безопасная последовательность для подготовленного релиза:

1. Создать отдельную public-ветку без примесей приватной структуры.
2. Переписать историю public-ветки перед публикацией (force push разрешен).
3. Пушить только public-ветку.

Пример команд (выполнять осознанно):

```bash
cd /path/to/project

git checkout -b public/v0.4

# Оставить в ветке только публичную папку и минимальные служебные файлы.
# Конкретные шаги зависят от текущего состояния рабочего дерева.

# После финальной проверки выполнить публикацию:
git push --force-with-lease origin public/v0.4
```

## Checklist перед push

- [ ] `v0.4-public` обновлен из рабочей версии
- [ ] runtime sample-файлы приведены к эталону
- [ ] secret/privacy scan не нашел совпадений
- [ ] тесты проходят
- [ ] diff содержит только ожидаемые публичные изменения
